# 客户 Chat 会话与身份体验设计

## 背景

当前客户侧 `/chat` 会从 `localStorage` 自动恢复唯一一条旧会话。用户即使隔了很久再次进入，也会直接看到上次聊天；页面顶部还暴露客户标识输入框，右侧展示偏内部工作台风格的服务上下文。这让客户入口更像调试或坐席工具，而不像主流客户服务产品。

本次改造聚焦客户侧 Chat：

- 短期返回时自然续聊，长期返回时进入新咨询。
- 提供受身份保护的服务端历史咨询。
- 将客户标识与账户操作收进账户菜单。
- 保留未登录用户咨询通用规则的能力。

坐席工作台、运营看板和 Agent 编排逻辑不在本次范围内。

## 产品规则

### 进入 Chat

客户进入 `/chat` 时：

1. 前端检查是否存在有效的客户身份令牌。
2. 有效令牌存在时，请求该客户最近活跃会话。
3. 最近会话最后一条消息距当前时间不超过 2 小时，自动加载并继续该会话。
4. 最近会话最后一条消息距当前时间超过 2 小时，或客户没有历史会话，展示全新的咨询首页。
5. 没有有效令牌时，也展示全新的咨询首页，不自动恢复本地旧消息。

2 小时以服务端记录的最后一条消息时间为准。边界规则为“不超过 2 小时继续，超过 2 小时新开”。

### 新咨询

- 新咨询页展示欢迎语、常见问题入口和消息输入框。
- 首条消息发送时才创建服务端会话。
- 用户可随时点击“新咨询”，无论当前会话是否仍在 2 小时内，均立即返回空白咨询页。
- 点击“新咨询”只切换当前视图，不删除历史会话。

### 历史咨询

- 历史咨询入口位于右上角账户菜单。
- 点击后打开右侧抽屉，按最后消息时间倒序展示会话。
- 每条历史记录展示摘要、状态和最后消息时间。
- 点击历史记录后加载该会话消息，并允许客户继续发送消息。
- 客户只能读取和继续属于自身身份令牌的会话。

### 客户身份

- 未验证客户可发起通用规则咨询。
- 查看历史咨询或访问个人订单数据前，要求验证身份。
- 验证方式为“手机号 + 最近订单号”。
- 验证成功后签发客户身份 JWT，有效期 7 天。
- 主动退出、切换账户、令牌过期或令牌校验失败时，清除客户身份状态。
- 令牌过期不会阻止进入 Chat 或通用咨询，但个人数据功能需要重新验证。

### 页面结构

- 移除顶部直接可编辑的客户标识输入框。
- 右上角使用账户按钮：
  - 未登录时显示“验证身份”。
  - 已登录时显示脱敏手机号，并提供历史咨询、新咨询、切换账户和退出登录。
- 移除客户侧右栏中的“服务上下文”“当前进度”和“处理进度”面板。
- 主聊天区保持单列、聚焦的客户服务体验。
- AI 回复仍可展示对客户有价值的知识引用；内部 Agent 信息保持隐藏。

## 后端设计

### 数据模型

为 `Conversation` 增加：

- `last_message_at`: 最后一条客户、AI、系统或人工消息的服务端时间。

创建会话时设置 `last_message_at`。每次成功持久化消息时同步更新该字段。历史列表与最近活跃会话均按该字段排序。

现有 `customer_ref` 继续作为会话与客户身份的关联字段。客户令牌中的身份声明必须由验证接口生成，前端不能自行指定其他客户来访问历史。

### 客户验证

新增客户验证接口：

`POST /customer-auth/verify`

请求：

```json
{
  "phone": "13800000001",
  "recent_order_id": "20260531002"
}
```

验证流程：

1. 通过业务系统客户接口按手机号查找客户。
2. 校验订单属于该客户。
3. 校验订单是该客户最近订单。若存在同一时间订单，以业务系统稳定排序后的第一条为准。
4. 验证成功后签发有效期 7 天的客户 JWT。

响应包含客户令牌、脱敏手机号和到期时间。错误响应不区分“手机号不存在”与“订单不匹配”，避免泄露客户信息。

### 客户令牌

客户 JWT 与坐席 JWT 使用同一签名配置，但通过明确的角色和声明区分：

- `role`: `customer`
- `customer_ref`: 经过验证的客户手机号或稳定客户标识
- `exp`: 7 天后

坐席接口继续要求坐席或管理员角色。客户接口只接受客户角色，不能复用坐席权限。

### 客户会话接口

新增只面向已验证客户的接口：

- `GET /customer/conversations`
  - 返回当前客户的历史会话，按 `last_message_at` 倒序。
- `GET /customer/conversations/recent`
  - 返回最近会话及 `should_resume`。
  - `should_resume` 由服务端根据最后消息时间与 2 小时规则计算。
- `GET /customer/conversations/{conversation_id}/messages`
  - 仅在会话属于当前客户时返回消息。

所有客户会话接口从 JWT 读取客户身份，不接受用于越权查询的 `customer_ref` 参数。

### 发送消息

现有 `/chat` SSE 保留未登录通用咨询能力，并扩展为可选接收客户 Bearer token：

- 有有效客户令牌时，服务端从令牌确定 `customer_ref`，忽略请求体中的客户身份值。
- 继续已有会话时，服务端校验该会话属于当前客户。
- 未登录时创建匿名或临时客户会话；涉及个人订单数据的工具调用前返回身份验证要求。
- 每次持久化消息后更新 `Conversation.last_message_at`。

为兼容现有调用，第一版保留请求体中的 `customer_ref`；已登录请求必须以令牌身份为准。

## 前端状态设计

客户 Chat 状态拆分为：

- `customerSession`: 客户令牌、脱敏手机号、过期时间。
- `activeConversation`: 当前会话 ID 与消息。
- `history`: 服务端历史会话列表。
- `viewMode`: `new` 或 `conversation`。

不再使用 `chat_messages` 和 `chat_conversation_id` 作为跨会话恢复的可信来源。它们在迁移时清除，避免旧版本缓存再次覆盖服务端判断。

页面初始化：

1. 恢复并校验客户令牌。
2. 无有效令牌时进入 `new`。
3. 有有效令牌时请求最近会话。
4. `should_resume=true` 时加载消息并进入 `conversation`。
5. 否则进入 `new`。

历史抽屉按需加载。身份验证成功后重新执行最近会话判断。

## 错误与边界处理

- 身份令牌过期或接口返回 401：清除客户身份，保留当前通用咨询视图，个人功能弹出重新验证。
- 最近会话接口失败：降级为新咨询页，不恢复本地旧会话。
- 历史列表加载失败：抽屉内展示重试状态，不影响当前聊天。
- 历史会话已不存在或不属于当前客户：返回 404，前端提示会话不可访问并返回新咨询页。
- 验证失败：使用统一提示“手机号或最近订单号不匹配”。
- 用户打开历史会话后继续发送消息，该消息会更新 `last_message_at`，之后 2 小时内再次进入将自动续聊该会话。
- 人工消息和系统消息同样更新 `last_message_at`，避免人工刚回复后客户返回却被错误引导到新咨询。

## 测试范围

### 后端

- 手机号与最近订单号匹配时签发客户令牌。
- 非最近订单、其他客户订单和不存在数据均验证失败。
- 客户令牌有效期为 7 天，过期令牌被拒绝。
- 客户只能读取自己的历史会话和消息。
- 最近消息不超过 2 小时时 `should_resume=true`。
- 最近消息超过 2 小时时 `should_resume=false`。
- 客户、AI、系统和人工消息都会更新 `last_message_at`。
- 已登录发送消息时不能通过请求体伪造其他客户身份。

### 前端

- 无令牌进入时展示新咨询页。
- 有效令牌且最近消息在 2 小时内时自动恢复最近会话。
- 最近消息超过 2 小时时展示新咨询页。
- 点击“新咨询”清空当前视图但保留历史。
- 历史抽屉只在验证后可访问，并能加载选中的会话。
- 令牌过期时清除账户状态并要求重新验证。
- 顶部不再直接展示客户标识输入框，右侧内部进度面板不再出现。

## 非目标

- 接入短信验证码或第三方统一登录。
- 允许客户管理多个身份或合并多个手机号的历史。
- 删除、归档、搜索或重命名历史会话。
- 修改坐席工作台的会话列表和权限模型。
- 改造 Agent 的通用问答、工具编排或人在环路策略。
